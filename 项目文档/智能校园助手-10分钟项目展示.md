# 智能校园助手 - 10分钟项目展示

---

## 1. 项目背景与目标

### 项目背景
- 校园生活服务需求分散，缺乏统一平台
- 学生对二手交易、餐饮订单、课程管理有强烈需求
- 传统校园服务效率低下，用户体验差

### 项目目标
- 构建一站式校园生活服务平台
- 整合二手交易、餐饮订单、课程表管理、聊天功能
- 提升学生校园生活效率和便利性

### 核心价值
- 降低学生生活成本（二手交易）
- 提高生活效率（在线点餐、课程管理）
- 增强校园社交（聊天功能）

---

## 2. 技术选型

### 前端技术栈
- **框架**: Vue 2
- **UI组件库**: Element UI
- **状态管理**: Vuex
- **路由**: Vue Router
- **HTTP请求**: Axios

### 后端技术栈
- **框架**: Django 3.2
- **API框架**: Django REST Framework
- **数据库**: SQLite
- **认证**: JWT

### 技术架构
- **前后端分离**：前端负责展示，后端提供API
- **RESTful设计**：统一的API接口规范
- **模块化开发**：各功能模块独立，便于扩展

---

## 3. 核心功能展示

### 3.1 二手交易模块

#### 核心功能
- **商品发布与管理**：用户可发布、编辑、删除二手商品
- **智能商品分配**：自动分配商品到真实用户（ershou1, ershou2, ershou3）
- **发布者显示**：商品列表清晰展示发布者信息
- **智能按钮控制**：
  - 自己发布的商品：不显示"联系卖家"按钮
  - 他人发布的商品：显示"联系卖家"按钮

#### 技术亮点
- 商品均匀分配算法
- 智能按钮显示逻辑
- 响应式商品列表

### 3.2 聊天功能模块

#### 核心功能
- **实时聊天**：买卖双方在线沟通
- **微信风格UI**：时间显示在消息上方，符合用户习惯
- **消息类型支持**：文本、图片、文件类型消息
- **交易消息列表**：集中管理与商品相关的聊天

#### 技术亮点
- 对话与消息模型设计
- 微信风格UI优化
- 消息类型扩展机制

### 3.3 其他功能
- **课程表管理**：添加、编辑、查看课程
- **校园点餐**：浏览餐厅、在线点餐、订单管理
- **个人中心**：用户信息管理、设置

## 4. 架构设计

### 系统架构图
```
+------------------+     +------------------+     +------------------+
|    前端应用      |     |    后端API       |     |    数据库        |
|  Vue 2 + Element | <-->| Django REST      | <-->|   SQLite         |
|      UI          |     |  Framework       |     |                  |
+------------------+     +------------------+     +------------------+
```

### 模块划分
- **用户管理**：注册、登录、权限控制
- **二手交易**：商品发布、浏览、交易
- **聊天功能**：实时消息、对话管理
- **餐饮订单**：餐厅浏览、在线点餐
- **课程表**：课程管理、提醒

### 数据流转
1. 用户发送请求 → 前端处理 → 调用API
2. 后端接收请求 → 业务逻辑处理 → 数据库操作
3. 数据库返回数据 → 后端处理 → 返回API响应
4. 前端接收响应 → 渲染页面 → 展示给用户

## 5. 实现难点与解决方案

### 难点1：二手商品分配
- **问题**：商品没有分配到实际用户，导致展示异常
- **解决方案**：
  - 编写Python脚本，创建3个真实用户
  - 实现商品均匀分配算法
  - 自动更新商品发布者信息

### 难点2：聊天UI优化
- **问题**：聊天界面时间显示不符合用户习惯
- **解决方案**：
  - 优化消息气泡样式
  - 调整时间显示位置到消息上方
  - 实现微信风格布局

### 难点3：前后端分离架构
- **问题**：前后端数据交互复杂
- **解决方案**：
  - 采用JWT认证机制
  - 设计RESTful API
  - 使用Axios统一处理HTTP请求

### 难点4：人机协作开发
- **问题**：如何合理使用大模型辅助开发
- **解决方案**：
  - 建立"用户主导、大模型辅助"协作模式
  - 严格审查生成代码
  - 实现增量改进
